# Fortune Teller Game Requirements Baseline
# Last updated: 2026-05-15 (FortuneFlowController, judge prose, spirit on draw; see refinements-changes.md)

[Game Design Requirements]
- Theme: carnival fortune telling with supernatural demon interference
- Narrative stake: player must keep carnival alive or face catastrophic consequence
- Card system:
  - Deck target size: 30 cards
  - Starter set: 10 cards
  - Card fields: title, image, description, hidden theme, optional bias
  - Hidden themes: Greed, Vanity, Chaos, Power
- Reading duel:
  - Player gives positive interpretation
  - Demon AI gives negative interpretation
  - If player response is negative, demon agrees and judge phase is skipped
  - Demon delivery (prototype): optional two-pass generation — (1) structured JSON outline validated in-game, (2) spoken curse bound to that outline; if outline is missing or invalid, fall back to single-pass demon prompt (no hard failure)
- Judge system (shipped prototype):
  - Winner decided by deterministic FortuneDuelRubric (word lists, card morals, themes, magical energy)
  - Optional Ollama pass writes tent verdict prose only (does not change winner)
  - Legacy JSON judge available in TarotReadingDuelPipeline for experiments

[Gameplay Systems]
- Fortune teller reading loop
- Carnival management loop
- Demon dance battle loop (DDR style)
- Resource model:
  - gullible customers
  - money quota
  - reputation
  - magic energy
  - time
- Optional helper systems:
  - card hint book
  - customer interviews before readings

[Technical Requirements]
- Engine: Unity
- AI Runtime: Ollama (local LLM inference)
- AI Roles: Demon Interpreter, Judge Arbiter (and optional player-side assistant)
- Networking: HTTP integration from Unity to Ollama API
- Demon two-pass (when enabled in Inspector on `DemonTarotReader` / `TarotReadingDuelPipeline`):
  - Pass 1: model returns JSON only; parser extracts object between first `{` and last `}`; themes constrained to greed | vanity | chaos | power (case-insensitive)
  - Pass 2: model receives canonical serialized outline + spread context; output is the player-facing curse text
  - Extra latency: two sequential generate calls when two-pass is on
- Reliability:
  - request timeout
  - retry policy
  - safe fallback output
- UX:
  - loading indicator while waiting for LLM response
  - deterministic parse format for model output (gate/judge JSON; demon outline JSON when two-pass is used)

[Quality Requirements]
- Clear role-constrained prompts to minimize AI drift
- Logging for every interpretation round
- Reproducible test scenarios using fixed card sets
- Modular architecture to support future card and story expansion
