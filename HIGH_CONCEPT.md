# High Concept — Fortune Teller vs Spirit

## Elevator pitch

You are a tenth-generation carnival fortune teller, bound by family pacts to a booth that must keep paying its debts. Each customer wants a hopeful reading from the cards—but a **spirit** (demon) whispers a competing curse. The **tent** weighs who spoke truer to the draw. Win favor, keep customers, and survive the season.

## Core fantasy

- **Tone:** Dark carnival, theatrical, slightly comic card captions—but readings are abstract (themes and morals), not slapstick retellings of card jokes.
- **Player skill:** Word choice (theme vocabulary, hopeful vs harsh tone), reading the spread’s moral lean (Good / Neutral / Bad), and spending **magical energy** to bias the duel.
- **Opposition:** The spirit is not fair; it wants the customer’s despair. The player must sound genuinely helpful.

## The LLM’s role in the product

Three distinct uses of a **local** language model (Ollama):

| Role | Purpose | Decides outcome? |
|------|---------|------------------|
| **Spirit** | Writes the five-sentence curse after cards are drawn | No |
| **Judge (prose)** | Announces the settled duel to the customer in tent voice | No |
| **(Legacy / dev)** | Optional gate + old JSON judge in `TarotReadingDuelPipeline` | Gate can skip counter-read; JSON judge is experimental |

**Production scoring** is **rule-based** (`FortuneDuelRubric`): word lists, card morals, theme echo per card, tone, magical energy. The judge LLM only **narrates** facts already computed—it cannot change the winner.

The player’s fortune is **typed by the human** (not generated by the LLM in the main tent flow).

## Why a local model (Ollama)?

1. **Cost and iteration** — Unlimited playtests during development without per-token API billing.
2. **Privacy / offline demos** — Readings stay on the machine running the game; suitable for lab machines and exhibition builds.
3. **Latency tradeoff accepted** — Spirit and judge prose take seconds; design hides this with “thinking” UI and starting the spirit as soon as cards are drawn.
4. **Deterministic game layer** — Designers own win/loss math in C#; the LLM supplies flavor, not authority.
5. **Course / prototype fit** — HTTP to `localhost:11434` is easy to wire from Unity (`OllamaClient`) and document.

Cloud APIs remain an option later (swap `OllamaClient` endpoint); the architecture already treats the LLM as an HTTP service.

## Win / lose stakes (prototype)

- **Customers** — Teller wins the duel → +1 customer; spirit wins → −1 (game over at 0).
- **Magical energy** — Run resource (0–100); committed on **Read Fortune** for duel scoring; full meter can guarantee a teller win.
- **Verdict** — Player must **Make Judgement** then **Accept Verdict** to apply results and start the next round.

## Target loop (shipped prototype)

1. **Main Menu** → start game  
2. **Draw cards** (three-card spread) → **spirit LLM starts**  
3. Player writes and submits **Read Fortune** (commits energy)  
4. **Make Judgement** → rubric scores instantly → **judge prose LLM** (optional)  
5. **Accept Verdict** → resources update, next customer  

## Out of scope (current build)

- Full carnival management sim, demon dance battle, customer interview system (described in design docs, not fully implemented).
- Player fortune generation by AI in the main scene (dev harness `TarotReadingSmokeTest` only).
